## 环境准备

### 1. 安装必要软件

| 软件 | 版本要求 | 下载地址 | 说明 |
|------|----------|----------|------|
| Node.js | 18+ | https://nodejs.org/ | 选择 LTS 版本 |
| pnpm | 8+ | 终端运行 `npm install -g pnpm` | 包管理器 |
| VS Code | 最新版 | https://code.visualstudio.com/ | 代码编辑器 |
| Git | 最新版 | https://git-scm.com/ | 版本控制 |

### 2. 推荐的 VS Code 扩展

安装以下扩展以获得最佳开发体验：

- **GitHub Copilot** - AI 代码助手（需要订阅）
- **ESLint** - 代码检查
- **Prettier** - 代码格式化
- **TypeScript Vue Plugin (Volar)** - TypeScript 支持
- **Tailwind CSS IntelliSense** - Tailwind 智能提示

### 3. 克隆并安装项目

```bash
# 克隆项目
git clone <repository-url>
cd thingsvis

# 安装依赖（首次运行需要等待几分钟）
pnpm install
```

## AI 辅助开发工作流

### 使用 SpecKit 命令开发（推荐）

ThingsVis 采用 **Spec-Kit 规格驱动开发**模式。在 VS Code 中使用 Copilot Chat，可以通过 `/speckit.*` 系列命令快速完成从需求到实现的全流程， 其他cli也支持，比如claude cli等可以从参考https://github.com/github/spec-kit/tree/main。

#### SpecKit 命令速查

| 命令 | 用途 | 说明 |
|------|------|------|
| `/speckit.constitution` | 建立项目原则 | 创建项目的开发规范和指导原则 |
| `/speckit.specify` | 创建需求规格 | 描述要构建什么，关注 What 和 Why |
| `/speckit.plan` | 制定技术方案 | 提供技术栈和架构选择 |
| `/speckit.tasks` | 分解任务清单 | 从实现计划创建可执行的任务列表 |
| `/speckit.implement` | 执行实现 | 按计划执行所有任务，构建功能 |

#### 完整开发流程示例

**Step 1: 建立项目原则**（首次开发或新项目时）

```
/speckit.constitution 创建关注代码质量、测试标准、用户体验一致性和性能要求的开发原则
```

**Step 2: 创建需求规格**

描述你要构建的功能，关注"做什么"和"为什么"，不需要关心技术细节：

```
/speckit.specify 构建一个数据源管理功能，用户可以添加、编辑、删除数据源。
数据源支持静态数据、HTTP API、WebSocket 三种类型。
每个组件可以绑定一个或多个数据源，通过表达式获取数据。
```

**Step 3: 制定技术实现计划**

提供你的技术栈和架构选择：

```
/speckit.plan 使用 React 18 + TypeScript 实现。
数据源配置使用 Zod 进行类型验证。
状态管理使用 Zustand。
遵循项目现有的 packages 目录结构。
```

**Step 4: 分解任务清单**

```
/speckit.tasks
```

AI 会自动将实现计划分解为可执行的任务列表，保存在 `specs/xxx/tasks.md` 中。

**Step 5: 执行实现**

```
/speckit.implement
```

AI 会按照任务清单逐步实现功能，你可以在过程中进行审查和调整。

> 💡 **提示**：SpecKit 生成的规格文档会保存在 `specs/` 目录下，方便团队成员查阅和 AI 理解上下文。

### 手动使用规格文档开发

如果不使用 SpecKit 命令，也可以手动参考规格文档进行开发。

#### 工作流程

```
1. 阅读规格文档 → 2. 让 AI 理解上下文 → 3. AI 生成代码 → 4. 验证测试 → 5. 提交代码
```

#### 步骤详解

**Step 1: 找到相关的规格文档**

规格文档位于 `specs/` 目录，每个功能都有独立的文件夹：

```
specs/
├── 001-core-data-protocol/     # 核心数据协议
├── 001-plugin-ecosystem/       # 插件生态系统
├── 002-l0-kernel-engine/       # 内核引擎
├── 003-canvas-kernel/          # 画布内核
├── 004-load-registry-components/ # 组件注册加载
├── 005-studio-kernel-integration/ # Studio 集成
└── 006-multi-source-data/      # 多数据源
```

每个规格文件夹包含：
- `spec.md` - 功能规格说明
- `plan.md` - 实现计划
- `quickstart.md` - 快速开始
- `tasks.md` - 任务清单
- `contracts/` - API 契约定义

**Step 2: 将规格文档提供给 AI**

在 VS Code 中使用 Copilot Chat，可以这样操作：

```
方法一：直接附加文件
1. 打开 Copilot Chat (Ctrl+Shift+I)
2. 点击 📎 按钮附加相关的 spec.md 文件
3. 输入你的需求

方法二：使用 @workspace
1. 输入 @workspace 让 AI 理解整个项目
2. 提问时引用具体的规格文件路径
```

**示例 Prompt：**

```
@workspace 请参考 specs/001-plugin-ecosystem/spec.md 的规范，
帮我创建一个新的 basic 类别下的 button 组件插件。
请遵循 docs/vis-cli/component-development.md 中的目录结构规范。
```

**Step 3: 使用任务清单跟踪进度**

每个 spec 目录下的 `tasks.md` 是任务清单，格式如下：

```markdown
## 任务列表

- [x] 已完成的任务
- [ ] 待完成的任务
- [ ] 进行中的任务 (WIP)
```

你可以让 AI 帮你更新任务状态：

```
请帮我把 specs/003-canvas-kernel/tasks.md 中的 
"实现节点渲染" 任务标记为已完成
```

### 使用其他 AI 工具

#### Claude / ChatGPT

如果使用 Claude 或 ChatGPT，需要手动提供上下文：

1. **复制规格文档内容**到对话中
2. **说明项目技术栈**：
   ```
   这是一个 ThingsVis 项目，技术栈：
   - TypeScript 5.x (严格模式)
   - React 18
   - Rspack + Module Federation 2.0
   - LeaferJS (2D 画布渲染)
   - Zustand + Immer (状态管理)
   - Zod (运行时验证)
   ```
3. **提供相关代码文件**作为参考

#### Cursor IDE

Cursor 内置 AI 功能，使用方式类似：

1. 使用 `Ctrl+K` 选中代码后让 AI 编辑
2. 使用 `Ctrl+L` 打开聊天，可以 `@` 引用文件
3. 推荐将 `specs/` 和 `docs/` 目录添加到 Cursor 的上下文中

## 用 AI 快速解决问题

> 💡 **核心理念**：不要害怕问 AI，把 AI 当作你的"随时在线的资深同事"。关键是学会如何提问。

### 🎯 黄金提问法则

**提问三要素**：
1. **背景** - 你在做什么？
2. **问题** - 遇到了什么错误？
3. **期望** - 你想要什么结果？

#### ❌ 差的提问方式
```
代码报错了，帮我看看
```

#### ✅ 好的提问方式
```
我在开发 ThingsVis 的文本插件（plugins/basic/text），
运行 pnpm dev 时报错：
"Module not found: Can't resolve '@thingsvis/schema'"

我已经运行过 pnpm install，请帮我排查原因。
```

以下是针对不同场景的 Prompt 模板，直接复制使用：

### 🔧 AI 辅助调试技巧

#### 技巧 1：让 AI 逐步调试

当遇到复杂问题时，不要期望 AI 一次性解决，而是分步骤：

```
Step 1: "帮我分析这个错误信息可能的原因有哪些？"
Step 2: "让我们先验证第一个可能：xxx，我应该怎么做？"
Step 3: "排除了第一个原因，现在检查第二个可能..."
```

#### 技巧 2：提供上下文文件

在 VS Code Copilot Chat 中，使用以下方式提供更多上下文：

```
# 附加当前文件
打开要讨论的文件，AI 会自动获取上下文

# 附加相关文件
点击 📎 按钮，选择相关的文件

# 引用整个工作区
使用 @workspace 前缀

# 引用终端输出
使用 @terminal 前缀
```

#### 技巧 3：验证 AI 的建议

AI 不是 100% 正确的，请养成验证习惯：

```bash
# 1. AI 给出修改建议后，先做类型检查
pnpm typecheck

# 2. 运行开发服务器看是否正常
pnpm dev

# 3. 如果有测试，运行测试
pnpm test
```

### 🚨 紧急问题快速通道

当你完全卡住时，按以下顺序尝试：

| 步骤 | 操作 | 时间 |
|------|------|------|
| 1 | 把完整错误信息发给 AI，让它分析 | 2分钟 |
| 2 | 搜索 `specs/` 目录看是否有相关说明 | 3分钟 |
| 3 | 在项目中搜索类似的实现作为参考 | 5分钟 |
| 4 | 查看 Git 历史，看这部分代码最近有什么改动 | 3分钟 |
| 5 | 如果以上都不行，整理问题描述，求助同事 | - |

### 💡 AI 使用小贴士

1. **保持对话连续性**：同一个问题尽量在同一个对话中解决，AI 会记住上下文
2. **及时反馈**：如果 AI 的回答不对，告诉它哪里错了，它会修正
3. **学会说"为什么"**：不只是要答案，也问 AI 为什么这样做，帮助你学习
4. **保存有用的 Prompt**：好用的提问模板保存下来，下次直接用
5. **中英文混用没关系**：AI 都能理解，用你最舒服的方式表达

## 常见问题与解决方案

### 🔴 问题 1: `pnpm install` 失败

**症状**：安装依赖时报错

**解决方案**：

```bash
# 1. 清理缓存
pnpm store prune

# 2. 删除 node_modules 和 lock 文件
rm -rf node_modules
rm pnpm-lock.yaml

# 3. 重新安装
pnpm install
```

### 🔴 问题 2: 端口被占用

**症状**：启动时提示 `Port 3000 is already in use`

**解决方案**：

```bash
# Windows: 查找并关闭占用端口的进程
netstat -ano | findstr :3000
taskkill /PID <进程ID> /F

# 或者直接换一个端口（在对应的配置文件中修改）
```

### 🔴 问题 3: TypeScript 类型错误

**症状**：编辑器显示红色波浪线，`pnpm typecheck` 报错

**解决方案**：

```bash
# 1. 重新构建所有包（解决包之间的依赖问题）
pnpm -w build

# 2. 重启 VS Code 的 TypeScript 服务
# 按 Ctrl+Shift+P，输入 "TypeScript: Restart TS Server"
```

### 🔴 问题 4: 插件不显示

**症状**：创建的插件在 Studio 中看不到

**排查步骤**：

1. 确认插件已添加到 registry.json
2. 检查插件是否正确导出 `Main` 模块
3. 查看浏览器控制台是否有错误信息

### 🔴 问题 5: Git 冲突

**症状**：`git pull` 或 `git merge` 时报冲突

**解决方案**：

```bash
# 1. 查看冲突文件
git status

# 2. 在 VS Code 中打开冲突文件，会看到类似这样的标记：
# <<<<<<< HEAD
# 你的代码
# =======
# 别人的代码
# >>>>>>> branch-name

# 3. 选择保留哪段代码（或合并两者），删除标记符号

# 4. 标记冲突已解决
git add <冲突文件>
git commit -m "resolve: 解决合并冲突"
```

## Git 提交规范

### Commit Message 格式

我们采用 **Conventional Commits** 规范：

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

### Type 类型说明

| Type | 说明 | 示例 |
|------|------|------|
| `feat` | 新功能 | `feat(plugin): 添加按钮组件` |
| `fix` | 修复 Bug | `fix(kernel): 修复状态同步问题` |
| `docs` | 文档更新 | `docs(readme): 更新安装说明` |
| `style` | 代码格式（不影响功能） | `style: 格式化代码` |
| `refactor` | 重构（不是新功能也不是修复） | `refactor(ui): 重构组件结构` |
| `perf` | 性能优化 | `perf(canvas): 优化渲染性能` |
| `test` | 添加测试 | `test(kernel): 添加单元测试` |
| `chore` | 构建/工具变动 | `chore: 更新依赖版本` |
| `revert` | 回滚提交 | `revert: 回滚 feat(xxx)` |

### Scope 范围说明

| Scope | 说明 |
|-------|------|
| `kernel` | thingsvis-kernel 包 |
| `schema` | thingsvis-schema 包 |
| `ui` | thingsvis-ui 包 |
| `utils` | thingsvis-utils 包 |
| `studio` | apps/studio 应用 |
| `preview` | apps/preview 应用 |
| `plugin` | 插件相关 |
| `cli` | vis-cli 工具 |
| `docs` | 文档 |
| `deps` | 依赖更新 |

### 提交示例

```bash
# ✅ 好的提交信息
git commit -m "feat(plugin): 添加基础文本组件"
git commit -m "fix(kernel): 修复节点删除后状态未更新的问题"
git commit -m "docs(readme): 添加快速开始指南"
git commit -m "refactor(studio): 重构工具栏组件结构"

# ❌ 不好的提交信息
git commit -m "更新代码"
git commit -m "fix bug"
git commit -m "完成功能"
```

### VS Code 提交技巧

1. 在源代码管理面板（Ctrl+Shift+G）写提交信息
2. 可以安装 **Conventional Commits** 扩展辅助填写

## 分支协作规范

### 分支命名规则

```
<type>/<issue-number>-<short-description>
```

| 分支类型 | 命名格式 | 示例 |
|----------|----------|------|
| 功能开发 | `feature/<issue>-<desc>` | `feature/123-add-button-plugin` |
| Bug 修复 | `fix/<issue>-<desc>` | `fix/456-node-sync-error` |
| 文档更新 | `docs/<desc>` | `docs/update-readme` |
| 重构 | `refactor/<desc>` | `refactor/kernel-structure` |
| 紧急修复 | `hotfix/<desc>` | `hotfix/critical-crash` |

### 分支工作流

```
main (主分支，保护分支)
  │
  ├── develop (开发分支)
  │     │
  │     ├── feature/xxx (功能分支)
  │     ├── fix/xxx (修复分支)
  │     └── ...
  │
  └── release/v1.0.0 (发布分支)
```

### 日常开发流程

**补充说明**： 如果采用spec-kit规范每个需求AI会自动在/spec目录下新建一个需求目录，并对于的创建一个新的分支，开发小功能和bug修复等可以采用下面的提交方式。

```bash
# 1. 确保在最新的 develop 分支
git checkout develop
git pull origin develop

# 2. 创建新的功能分支
git checkout -b feature/123-add-button-plugin

# 3. 开发过程中，定期提交
git add .
git commit -m "feat(plugin): 实现按钮基础结构"

# 4. 推送到远程
git push origin feature/123-add-button-plugin

# 5. 在 GitHub 上创建 Pull Request，请求合并到 develop
```

### Pull Request 规范

创建 PR 时，请遵循以下模板：

```markdown
## 📋 概述
简要描述这个 PR 做了什么

## 🔗 关联 Issue
Closes #123

## 📝 改动内容
- [ ] 添加了 xxx 功能
- [ ] 修改了 xxx 文件
- [ ] 更新了 xxx 文档

## 🧪 测试
描述如何测试这些改动

## 📸 截图（如果有 UI 变化）
```

### 代码评审要求

- 至少 1 人批准后才能合并
- 所有 CI 检查必须通过
- 冲突必须解决
- Commit 信息符合规范
